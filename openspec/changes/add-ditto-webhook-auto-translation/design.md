## Context

Ditto emits webhook events when base text or variant text changes. The service needs to infer the changed source locale, translate from that source text, and update only the other configured locales through Ditto's Text Items API. Translation must use the local codex-lb OpenAI-compatible server running from `~/Desktop/codex-lb`, with the API key generated by that service.

The current repository has no application implementation yet, so this change introduces a small Python service, its configuration, and test coverage. The service will be called by Ditto over HTTP and will call both codex-lb and Ditto as outbound dependencies.

## Goals / Non-Goals

**Goals:**

- Accept Ditto base and variant text change webhook payloads.
- Detect the source locale from the changed Ditto locale and never overwrite that locale during the same event.
- Translate and update all remaining configured locales.
- Prevent duplicate processing of the same webhook delivery.
- Prevent cascades from webhook events caused by this service's own Ditto updates.
- Retry transient translation and Ditto update failures.
- Log processing outcomes without exposing API keys or translated text payloads.
- Keep Ditto locale and variant developer IDs configurable.

**Non-Goals:**

- Build a UI for managing translations or reviewing machine output.
- Store translated text history beyond short-lived loop-prevention state.
- Infer locale mappings from Ditto automatically.
- Support arbitrary localization workflows outside configured Ditto base/variant locales.

## Decisions

- Use FastAPI as the webhook runtime. It provides a small HTTP surface, async request body handling, and simple deployment with uvicorn. A pure script or worker would still need an HTTP adapter, so FastAPI keeps the boundary explicit.
- Use synchronous clients behind `anyio.to_thread.run_sync`. Ditto and OpenAI-compatible SDK calls are straightforward sync operations; running the service logic in a worker thread avoids blocking the event loop while keeping the implementation simple.
- Use SQLite for durable idempotency and outbound update memory. In-memory state would fail across restarts, while a separate queue/database would be premature for this small service.
- Track webhook events with statuses. `succeeded` and `skipped` events are treated as completed duplicates; active `in_progress` redeliveries return a retryable response; `failed` and stale `in_progress` events may be retried so Ditto redelivery is useful.
- Track self-generated updates by `(project_id, developer_id, locale, translated_text_hash)` with a TTL. Ditto can echo the service's own target-locale write as another webhook; matching the exact locale and text lets the service skip that echo without ignoring real designer edits.
- Verify Ditto signatures by default. Production should reject unsigned webhook requests; local development can explicitly opt into unsigned events if needed.
- Treat malformed webhook payloads and non-rate-limited Ditto 4xx responses as non-retryable skipped outcomes so permanent failures do not create webhook/API retry loops.
- Use codex-lb through the OpenAI Python SDK with `CODEX_LB_BASE_URL`, `CODEX_LB_API_KEY`, and `TRANSLATION_MODEL`. This matches the local Docker server and keeps translation provider wiring replaceable.
- Send Ditto update requests through `PATCH /textItems`, including the webhook `projectId`, omitting `variantId` for the base locale, and including the mapped variant developer ID for variant locales.

## Risks / Trade-offs

- Ditto webhook `textItemId` must correspond to the developer ID expected by `PATCH /textItems` -> If a workspace uses a different identifier, the service will need an additional lookup step before updates.
- A designer could intentionally set a target locale to the exact same text just generated by the service within the TTL -> The event may be treated as self-generated and skipped; the TTL is configurable and short-lived.
- SQLite is local to one service instance -> Horizontal deployment requires moving idempotency and outbound-update state to a shared store.
- Machine translation can damage placeholders or tone -> The prompt explicitly preserves placeholders/tags and tests focus on routing, but human QA remains outside scope.
- Ditto Authorization scheme details can vary by token format -> The service sends the configured `DITTO_API_TOKEN` verbatim as the `Authorization` header so deployment can provide the exact required value.

## Migration Plan

1. Configure Ditto variant developer IDs and the base locale in environment variables.
2. Start the codex-lb Docker server from `~/Desktop/codex-lb` and generate an API key from its dashboard.
3. Deploy the FastAPI service with Ditto API token, Ditto webhook signing key, and codex-lb configuration.
4. Register the Ditto webhook endpoint and test with a non-production Ditto project.
5. Roll back by disabling the Ditto webhook; no Ditto schema migration is required.
